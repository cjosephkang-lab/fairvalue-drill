# フェアバリューアプリ：72の法則・買収判断チートシート追加 設計書

作成日: 2026-06-29
対象アプリ: `fairvalue-drill`（React + Vite + Tailwind の数値ドリルアプリ）

## 背景・目的

CEO（姜彰周）が「買収判断 瞬時計算チートシート」（利回り = 年間FCF ÷ 買収金額、72の法則、割引率比較）を
フェアバリューアプリで **練習できる** ようにし、かつ **いつでも立ち戻れる読み物** としても残したい。
さらにこの買収判断フレームワークを AI の長期メモリにも保存し、今後の M&A・ロールアップ判断で再利用できるようにする。

## スコープ

### やること（今回のセッション）
1. ドリル4問追加（Lv.7〜10）— チートシートの3ステップを分解
2. 「チート」タブ追加 — チートシート全文を常設の読み物として表示
3. 履歴保存の土台化 — localStorage アクセスを薄い1ファイルに閉じ込め、将来クラウド差し替え可能に（動作は不変）
4. AI メモリ保存 — 買収判断フレームワークを `memory/` に1ファイル + MEMORY.md に1行
5. GitHub Pages にデプロイ（自分用・URLは人に配らない）

### やらないこと（別案件）
- **B: 個人プロフィール切替**（端末内で名前ごとに履歴を分ける）— 人が増える兆しが出てから
- **C: クラウドログイン + 顧客配布**（Firebase Auth / Firestore / セキュリティルール / 管理画面）
  - 配布が現実に動き出すタイミングで `brainstorming → spec → 実装` を別途一周する
  - 今回の「履歴保存の土台化」は、このCへ上げるときの差し替えコストを下げるための布石

## アーキテクチャ方針

既存パターンを壊さない。`Problem` 型は変更しない。新UIは最小限（タブ切替のみ）。

### 全体構造（追加・変更点）

```
App.tsx                     ← タブ状態(useState)を追加。「ドリル / チート」を切替
├ ドリルビュー（既存のまま：ProgressBar + ProblemCard / 完了画面）
└ チートビュー（新規：CheatSheet）

src/data/problems.ts        ← problems 配列に Lv.7〜10 を追記（型変更なし）
src/data/cheatsheet.ts      ← 新規。チートシートの内容をデータとして保持
src/components/CheatSheet.tsx ← 新規。cheatsheet.ts を表示する読み物専用コンポーネント（状態なし）
src/lib/storage.ts          ← 新規。localStorage の load/save を閉じ込める薄い層
src/hooks/useProgress.ts    ← 変更。直接の localStorage 参照を storage.ts 経由に置換（動作不変）
```

## セクション1：ドリル4問（Lv.7〜10）

`src/data/problems.ts` の `problems` 配列に既存の `Problem` 型のまま追記する。

| Lv | tag | question 概要 | 判定 |
|----|-----|--------------|------|
| Lv.7 | 利回り入門 | 売り手の言い値3億、年間FCF1,500万。利回りは何%？ | `answer: 5` |
| Lv.8 | 72の法則 | 利回り5%。72の法則で回収年数は約何年？ | `answer: 14.4`, `tolerance: 0.5` |
| Lv.9 | 割引率と比較 | 利回り5% vs あなたの割引率12%。買い=1／高い=0 で答えよ | `answer: 0` |
| Lv.10 | 商談まるごと | 言い値3億・FCF1,500万・割引率12%。利回り基準の妥当買値（FCF÷割引率）は？ | `answer: 12500`, `answer_min: 12000`, `answer_max: 13000` |

### 各問の計算根拠
- Lv.7: 1,500万 ÷ 30,000万 = 0.05 = **5%**
- Lv.8: 72 ÷ 5 = **14.4年**（誤差±0.5年を許容）
- Lv.9: 利回り5% < 割引率12% → 要求水準に届かない＝**高い（0）**
- Lv.10: 1,500万 ÷ 0.12 = **12,500万円**（=1.25億）。この値段までなら割引率12%を満たす。言い値3億は高すぎる。
  - 単位は既存に合わせ「万円」。範囲 12,000〜13,000 を正解とする。

### 設計判断（確定済み）
- Lv.9 は新UIを足さず、既存の数値入力で **1=買い / 0=高い** を入力させる。問題文・hint に明記。
- 各問の `explanation` / `concept` にチートシートの言葉をそのまま入れる
  （「売り手の言い値をFCFで割れ」「72÷利回り」「割引率＝これ以下なら投資しないライン」「妥当買値＝FCF÷割引率」）。
- 全問クリア画面（App.tsx 完了セクション）の文言に72の法則を1行追加。

## セクション2：「チート」タブ

### UI
`App.tsx` 最上部に2択タブ（ドリル / チート）を `useState<"drill" | "cheat">` で切替。
ルーティングライブラリは入れない。既存のダーク配色（`bg-[#0F1117]`, `border-white/10` 等）を踏襲。

### 内容（CEO提供のチートシート全文）
`src/data/cheatsheet.ts` に構造化データとして保持し、`CheatSheet.tsx` が描画：
- 核心の公式（利回り = 年間FCF ÷ 買収金額 / 回収年数 ≈ 72 ÷ 利回り(%)）
- 3ステップ（①利回り ②72の法則 ③割引率と比較）
- 72の法則 早見表（表：0.1%→720年 / 2%→36年 / 5%→14年 / 10%→7年★ / 15%→4.8年）
- 商談での動き方（6手順）
- 割引率の正体（中小・非上場10〜15% / 大企業5〜7%）
- 注意点（成長ゼロ・永続前提の近似 / 成長企業はもっと高評価 / 最終判断はExcel）
- 一行で覚える（「売り手の言い値をFCFで割れ。それが全ての入口。」）強調枠

### 実装方針
- `CheatSheet.tsx` は状態なしの純表示コンポーネント。
- 配色・角丸・枠線は `ProblemCard` のトークンを流用しデザイン統一。

## セクション3：履歴保存の土台化・AIメモリ・デプロイ

### 3-1. 履歴保存の土台化
- `src/lib/storage.ts` を新規作成。`loadProgress()` / `saveProgress()` の2関数だけを公開。
- 中身は現状の localStorage 実装をそのまま移植（キー・形式・動作すべて不変）。
- `useProgress.ts` は直接 localStorage を触らず storage.ts 経由にする。
- 目的：将来 C でクラウド化する際、storage.ts の中身だけを
  「ログイン中ユーザーのID配下の Firestore 読み書き」に差し替えれば済む状態にする。
- **このリファクタで既存の進捗データ・動作が変わってはならない**（回帰がないことをテストで担保）。

### 3-2. AIメモリ保存
- `memory/project-ma-valuation-72-rule.md` を新規作成（type: reference または project）。
  - 内容：利回り = FCF ÷ 買収金額、回収年数 ≈ 72 ÷ 利回り、割引率比較（買い/高い）、妥当買値 = FCF ÷ 割引率、
    割引率の目安（中小・非上場10〜15% / 大企業5〜7%）、近似の前提（成長ゼロ・永続）。
  - 既存メモリ `project-ai-keiei-migite-rollup-bdash` 等とリンク。
- `MEMORY.md` に1行ポインタを追加。

### 3-3. デプロイ（GitHub Pages・自分用）
- 既存設定で準備済み：`homepage`, `gh-pages` 依存, `deploy` スクリプト, `vite.config.ts` の `base: "/fairvalue-drill/"`,
  リモート `github.com/cjosephkang-lab/fairvalue-drill`。
- 手順：ローカル動作確認 → `/ui-screenshot` で自己確認 → `npm run deploy`。
- URLは人に配らない（自分用）。配布は別案件C。

## テスト方針

- **checkAnswer のユニットテスト**：Lv.7〜10 の各問について、正解値・境界（tolerance / range）・不正解が正しく判定されること。
  - 特に Lv.9（1/0判定）と Lv.10（範囲判定）。
- **storage.ts の回帰テスト**：リファクタ前後で load/save の入出力が同一であること（既存進捗が壊れない）。
- **ビルド**：`npm run build`（tsc + vite build）が通ること。
- **目視（UI）**：`/ui-screenshot` でドリルタブ・チートタブ・Lv.7〜10 の表示を自己確認。

## リスク・注意点

- Lv.9 の「1=買い / 0=高い」は直感的でない可能性 → 問題文・hint で明示し、解説でも補足する。
- storage.ts リファクタは「動作不変」が絶対条件。localStorage のキー名・JSON形状を変えない。
- GitHub Pages は公開URLになる（自分用でもURLを知れば誰でも開ける）。配布しないことで運用上の事故を避ける。
  履歴は端末内 localStorage のままなので、複数人が同URLを別端末で開いても互いの履歴は混ざらない
  （同一端末・同一ブラウザを共有した場合のみ混ざる）。
